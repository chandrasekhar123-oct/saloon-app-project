2. MODULE 1 — USER AUTHENTICATION & LOGIN
================================================================================

--------------------------------------------------------------------------------
TC-AUTH-001: Successful Login with Valid Credentials
--------------------------------------------------------------------------------
Test Case ID  : TC-AUTH-001
Title         : Successful login with valid credentials
Type          : Functional
Priority      : High
Preconditions : User account exists; app is accessible

Steps:
  1. Navigate to https://[app-url]/login
  2. Enter a registered email address in the Email field
  3. Enter the correct password in the Password field
  4. Click the Login button

Expected Result:
  User is redirected to the dashboard. A session token is generated and
  stored securely.

--------------------------------------------------------------------------------
TC-AUTH-002: Login with Invalid Credentials
--------------------------------------------------------------------------------
Test Case ID  : TC-AUTH-002
Title         : Login attempt with incorrect password
Type          : Functional + Security
Priority      : High
Preconditions : User account exists

Steps:
  1. Navigate to /login
  2. Enter a valid registered email
  3. Enter an incorrect password
  4. Click the Login button

Expected Result:
  Error message displayed: "Invalid email or password."
  No session token created. Account not locked on first attempt.

--------------------------------------------------------------------------------
TC-AUTH-003: Brute Force / Account Lockout
--------------------------------------------------------------------------------
Test Case ID  : TC-AUTH-003
Title         : Account lockout after repeated failed login attempts
Type          : Security
Priority      : Critical
Preconditions : User account exists; lockout policy configured

Steps:
  1. Navigate to /login
  2. Enter a valid email and incorrect password
  3. Repeat Step 2 five consecutive times
  4. Attempt login a sixth time with correct credentials

Expected Result:
  Account is temporarily locked. User receives a lockout notification.
  Correct credentials do not bypass the lock.

--------------------------------------------------------------------------------
TC-AUTH-004: Regression — Login After Password Reset
--------------------------------------------------------------------------------
Test Case ID  : TC-AUTH-004
Title         : User can login successfully after a password reset
Type          : Regression
Priority      : High
Preconditions : User has completed password reset flow

Steps:
  1. Complete password reset via email link
  2. Navigate to /login
  3. Enter email and the new password
  4. Click Login

Expected Result:
  User logs in successfully. Old password no longer works.

3. MODULE 2 — PAYMENTS & CHECKOUT
================================================================================

--------------------------------------------------------------------------------
TC-PAY-001: Successful Checkout with Valid Card
--------------------------------------------------------------------------------
Test Case ID  : TC-PAY-001
Title         : Complete checkout with a valid credit card
Type          : Functional
Priority      : Critical
Preconditions : User is logged in; cart has at least one item;
                payment gateway is in test mode

Steps:
  1. Add a product to the cart
  2. Navigate to /checkout
  3. Fill in shipping address form with valid data
  4. Enter test card number: 4242 4242 4242 4242, expiry 12/26, CVV 123
  5. Click Place Order

Expected Result:
  Order confirmation page displayed. Order ID generated. Confirmation email
  triggered. Payment captured successfully in gateway.

--------------------------------------------------------------------------------
TC-PAY-002: Checkout with Expired Card
--------------------------------------------------------------------------------
Test Case ID  : TC-PAY-002
Title         : Payment failure with an expired card
Type          : Functional
Priority      : High
Preconditions : User is logged in; cart is not empty

Steps:
  1. Navigate to /checkout
  2. Fill in valid shipping details
  3. Enter expired card number with a past expiry date
  4. Click Place Order

Expected Result:
  Payment declined. Error message: "Your card has expired. Please use a
  different payment method." Order is NOT placed.

--------------------------------------------------------------------------------
TC-PAY-003: SQL Injection in Payment Form
--------------------------------------------------------------------------------
Test Case ID  : TC-PAY-003
Title         : Security — SQL injection attempt on payment form fields
Type          : Security
Priority      : Critical
Preconditions : User is on the checkout page

Steps:
  1. Navigate to /checkout
  2. In the Cardholder Name field, enter: '; DROP TABLE orders;--
  3. Fill in other fields with valid test data
  4. Click Place Order

Expected Result:
  Input is sanitized. No database error exposed. Appropriate validation
  error shown to user. Database tables remain intact.

--------------------------------------------------------------------------------
TC-PAY-004: Performance — Checkout Under Load
--------------------------------------------------------------------------------
Test Case ID  : TC-PAY-004
Title         : Checkout page response time under 200 concurrent users
Type          : Performance
Priority      : High
Preconditions : Load testing tool configured (e.g., JMeter / k6)

Steps:
  1. Simulate 200 concurrent users navigating to /checkout
  2. Each user adds one item to cart and initiates checkout
  3. Record response times and error rates over a 5-minute window

Expected Result:
  Average response time <= 2 seconds. Error rate < 1%.
  No server crashes or timeouts.

Type          : Functional

Title         : Security — XSS attempt via search bar
Type          : Security
Priority      : Critical
Preconditions : User is on the homepage

5. MODULE 4 — APIs & INTEGRATIONS
================================================================================

--------------------------------------------------------------------------------
TC-API-001: GET Products API — Valid Request
--------------------------------------------------------------------------------
Test Case ID  : TC-API-001
Title         : Fetch product list via API with valid auth token
Type          : Functional
Priority      : High
Preconditions : API server is running; valid bearer token available

Steps:
  1. Open API testing tool (e.g., Postman)
  2. Send GET /api/v1/products
  3. Include header: Authorization: Bearer <valid_token>
  4. Review response

Expected Result:
  HTTP 200 OK. Response body contains a paginated JSON array of products
  with correct schema.

--------------------------------------------------------------------------------
TC-API-002: API Request Without Auth Token
--------------------------------------------------------------------------------
Test Case ID  : TC-API-002
Title         : API returns 401 for unauthenticated request
Type          : Security
Priority      : Critical
Preconditions : API server is running

Steps:
  1. Send GET /api/v1/products WITHOUT any Authorization header
  2. Review the response status and body

Expected Result:
  HTTP 401 Unauthorized.
  Response body: {"error": "Authentication required"}
  No product data is returned.

--------------------------------------------------------------------------------
TC-API-003: API Performance Under Load
--------------------------------------------------------------------------------
Test Case ID  : TC-API-003
Title         : Product listing API response time under 500 requests/sec
Type          : Performance
Priority      : High
Preconditions : Load test tool configured

Steps:
  1. Use k6 or JMeter to send 500 requests/second to GET /api/v1/products
  2. Run for 3 minutes
  3. Monitor response times, throughput, and error rates

Expected Result:
  P95 response time <= 500ms. Error rate < 0.5%.
  No memory leaks or server crashes detected.
